This fork does not currently publish verified prebuilt images or binaries.
Build from `nstefanelli/homebox` so the fork's container, AI, icon, cataloging,
and integrations features are present. Images and binaries published by
`sysadminsmedia/homebox` are upstream builds and do not contain those features.

There are two supported local-build paths:

1. A Docker image built from this repository (recommended)
2. A native binary built from this repository

The application can be configured with environment variables. See the
[configuration reference](/en/quick-start/configure/) for the complete list.

## Docker

Install Docker, then clone this fork:

```bash
git clone https://github.com/nstefanelli/homebox.git
cd homebox
```

Build the image variant you want:

<Tabs>
    <TabItem label="Regular">
        ```bash
        docker build --pull --file Dockerfile --tag homebox-fork:local .
        ```
    </TabItem>
    <TabItem label="Rootless">
        ```bash
        docker build --pull --file Dockerfile.rootless --tag homebox-fork:local-rootless .
        ```
    </TabItem>
    <TabItem label="Hardened">
        ```bash
        docker build --pull --file Dockerfile.hardened --tag homebox-fork:local-hardened .
        ```
    </TabItem>
</Tabs>

The rootless and hardened images run as UID/GID `65532`. The hardened image is
distroless and does not contain a shell, utilities, or the MQTT client.

### Required API-key pepper

Generate the pepper once, store it with restrictive permissions, and reuse the
same file after every restart:

```bash
printf 'HBOX_AUTH_API_KEY_PEPPER=%s\n' "$(openssl rand -base64 48)" > .env
chmod 600 .env
```

The value must be at least 32 bytes. Rotating or losing it invalidates every
issued API key.

### Docker Run

<Tabs>
    <TabItem label="Regular">
        ```bash
        docker run -d --name homebox --restart unless-stopped \
          --publish 3100:7745 \
          --env-file .env \
          --env HBOX_OPTIONS_ALLOW_ANALYTICS=false \
          --volume homebox-data:/data \
          homebox-fork:local
        ```
    </TabItem>
    <TabItem label="Rootless">
        ```bash
        docker run -d --name homebox --restart unless-stopped \
          --publish 3100:7745 \
          --env-file .env \
          --env HBOX_OPTIONS_ALLOW_ANALYTICS=false \
          --volume homebox-data:/data \
          homebox-fork:local-rootless
        ```
    </TabItem>
    <TabItem label="Hardened">
        ```bash
        docker run -d --name homebox --restart unless-stopped \
          --publish 3100:7745 \
          --env-file .env \
          --env HBOX_OPTIONS_ALLOW_ANALYTICS=false \
          --volume homebox-data:/data \
          homebox-fork:local-hardened
        ```
    </TabItem>
</Tabs>

For a bind mount instead of the named volume, create the directory first. For
the rootless or hardened image, make it writable by the runtime user:

```bash
mkdir -p /path/to/homebox-data
chown -R 65532:65532 /path/to/homebox-data
```

Then replace `--volume homebox-data:/data` with
`--volume /path/to/homebox-data:/data`.

### Docker Compose

The checked-in `docker-compose.yml` builds the regular image, requires the
pepper from `.env`, and persists `/data` in a named volume. From the repository
checkout, start it with explicit build provenance:

```bash
HBOX_BUILD_COMMIT="$(git rev-parse HEAD)" \
HBOX_BUILD_TIME="$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
docker compose up -d --build
docker compose ps
```

For a separately built rootless or hardened image, create `compose.yml` with
the same secret and persistence contract:

```yaml title="compose.yml"
services:
  homebox:
    # Use homebox-fork:local-rootless or homebox-fork:local-hardened
    # when you built that variant.
    image: homebox-fork:local
    pull_policy: never
    restart: unless-stopped
    environment:
      HBOX_LOG_LEVEL: info
      HBOX_LOG_FORMAT: text
      HBOX_WEB_MAX_UPLOAD_SIZE: 10
      HBOX_OPTIONS_ALLOW_ANALYTICS: "false"
      HBOX_AUTH_API_KEY_PEPPER: ${HBOX_AUTH_API_KEY_PEPPER:?set HBOX_AUTH_API_KEY_PEPPER in .env}
    volumes:
      - homebox-data:/data
    ports:
      - "3100:7745"

volumes:
  homebox-data:
```

Start the service from the directory containing both files:

```bash
docker compose up -d
docker compose ps
```

Open `http://server.local.ip.address:3100`.

## Native binary

Native builds require Go `1.26.5`, Node.js `22` or newer, and pnpm `10.28.0`.
From a clone of this repository:

```bash
pnpm --dir frontend install --frozen-lockfile
pnpm --dir frontend build
cp -R frontend/.output/public/. backend/app/api/static/public/

mkdir -p build
(cd backend && go build -o ../build/homebox ./app/api)
```

Generate `.env` as described above, export it, and start the binary:

```bash
set -a
. ./.env
set +a
./build/homebox
```

Open `http://server.local.ip.address:7745`.

> [!NOTE]
> Docker is the validated production path for this fork. A native build uses
> the operating system and architecture of the machine on which `go build`
> runs.
